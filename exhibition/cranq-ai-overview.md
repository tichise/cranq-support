# Cranq — LLM の組み込み方と設計上の選択

生成AIなんでも展示会向け。LLM 開発者・実装者に向けて、Cranq の LLM 統合を **(1) 役割と統合** **(2) 設計空間における選択** **(3) 体験の手触り** **(4) やる／やらないの境界** の四つの切り口で解説します。

ネットワーク経路や鍵管理など API 呼び出しの実装プラミングは本資料の主題ではないため割愛します。

---

## 1. 役割と統合

LLM は **「振り返りのナレーター」** として置いている。アプリ内では「AI コーチ」と表記。

### 1.1 入力（LLM に何を見せているか）

- **当日のワークアウト**: 種目・距離・時間・標高ゲイン・平均/最大心拍・ペース（HealthKit 由来）
- **当日の天気**: 気温・降水・風など（Open-Meteo を端末から呼んで合流）
- **中期コンテキスト**: 週次／月次のトレンド、ストリーク日数
- **入れないもの**: 氏名・連絡先・端末識別子などの個人識別情報

意図的に **異なる軸（運動指標 × 気象 × 中期トレンド × 継続性）** を同時に渡している。LLM が最も得意とするのは **複数の横断した情報から傾向を掴む** ことなので、入力をヘテロに重ねるほど「直近3週はペースを上げたが心拍は抑えめ」のような **軸を跨いだ解釈** が引き出せる。

### 1.2 出力（LLM に何を書かせているか）

- **形式**: 自然言語の一段落（短）
- **配置**: 統計画面の「今日のサマリー」上に AI コーチからの一言として表示
- **個別性**: 本人の数値推移に紐づく文。「直近3週は安定」「次のロング走は心拍を抑えめに」のような具体まで踏み込む

## 2. 設計空間における選択 — なぜ「ナレーター型」か

フィットネスアプリに LLM を組み込むときの典型的な選択肢:

| 型 | 役割 | 主な難所 |
| --- | --- | --- |
| **コーチ型チャット** | 「次のメニューは？」と質問できる対話相手 | 評価軸が多次元、状態管理、品質保証 |
| **プランナー型** | ワークアウト計画を生成 | 未来推論のハルシネーション、怪我リスク |
| **エージェント型** | カレンダー・通知と連動して走る予定を組む | スコープ拡散、ツール呼び出しの信頼性 |
| **フォーム解析型** | 動画を見て姿勢を指摘 | マルチモーダル精度、誤指摘の体感被害 |
| **ナレーター型** ← Cranq | 走った後に意味付けの一段落を書く | スコープを閉じれば被害も小さい |

Cranq が **ナレーター型** を選んだ理由:

- **LLM が最も得意な「複数軸の横断統合」に乗せている**
  - 渡しているのは運動指標 × 気象 × 週次/月次トレンド × ストリーク
  - これら **異なる軸の数値群を跨いで傾向を掴み、自然文に落とす** ことこそ LLM 固有の得意分野
  - ルールベースでも「距離が伸びた」「心拍が高い」程度なら書けるが、複数軸を組み合わせた解釈（「ペース上昇 × 心拍抑制 × 3週連続 → フォーム改善が効いている可能性」）は組み合わせ爆発で書けない
- **被害コストが小さい**
  - 過去解釈なら間違っても「文章として変」で済む
  - プラン生成だと「無理なメニュー指示」で怪我のリスク
- **評価軸が単純化される**
  - ワークアウト1件 → 段落1つ、入出力が単発で回帰評価しやすい
  - 対話履歴も多段思考も持たないので状態管理の地獄に入らない
- **揺らぎが歓迎要素になる領域**
  - 振り返り用途では文体の多様性は欠点ではなく魅力

## 3. 体験の手触り

LLM がいることで実際に何が起きているか:

- **「数値」と「文」が同じ画面に同居する** — 統計画面の「今日のサマリー」に、数字と LLM の一段落が並んで表示される
- **複数軸を横断した傾向が文になる** — 運動指標 × 気象 × 週次/月次トレンド × ストリークが同時に入力に乗っているので、「雨の中でこの距離、しかも先週よりペースが安定」のように **異なる軸の組み合わせから傾向を掴む** タイプの解釈が出る。LLM が最も得意とする領域
- **文脈依存のトーンが乗る** — 雨の日／暑い日／休み明け／調子の良い日で、文の構え方が自動的に切り替わる
- **読むだけで振り返りが終わる** — 走り終わるたびに自分で意味付けする必要がなく、ユーザーは「読む」側に回れる
- **同じワークアウトでも文は毎回少し違う** — テンプレ応答にならない。多様性は振り返り用途では歓迎要素

## 4. やる／やらないの境界（採用判断）

| 判断 | 内容 | 理由 |
| --- | --- | --- |
| ✅ **やる** | 過去解釈／ナレーション | LLM の歩留まりが高い、被害コストが小さい |
| ✅ **やる** | 複数軸の横断入力（運動指標 × 気象 × 中期トレンド × ストリーク） | **複数の横断した情報から傾向を掴む** という LLM 固有の得意分野に賭ける |
| ❌ **やらない** | 未来プラン生成（「来週これを走れ」） | 運動指導領域でのハルシネーション被害を回避 |
| ❌ **やらない** | 対話ループ（チャット相手化） | 評価軸が多次元化、ナレーション役に絞ることで品質を保つ |
| ❌ **やらない** | エージェント化（ツール呼び出し・多段推論） | スコープ拡散を避け、「workout 1件 → 段落1つ」に閉じる |
| ⚠️ **割り切り** | 出力のばらつきを許容 | 振り返り用途では文体多様性はむしろ歓迎 |

---

## 5. 端的に（A4プレート補強・口頭用）

- **設計上の選択**: フィットネス × LLM の中で **ナレーター型** を選択。コーチ型チャット／プランナー／エージェントではない
- **入力の妙**: 運動指標 × 気象 × 週次/月次トレンド × ストリークを同時に渡す。**複数の横断した情報から傾向を掴む** という LLM 固有の得意分野に賭けている
- **スコープ**: 「ワークアウト1件 → 段落1つ」に閉じる。対話せず、計画もせず、エージェント化もしない
- **狙い**: LLM が最も得意な領域に乗せ、被害コストを小さく、評価軸を単純に保つ
