# Cranq — LLM の組み込み方と設計上の選択

LLM をフィットネスアプリに乗せる選択肢は複数あるが、Cranq は **能動的なエージェントではなく、走った後に意味付けの一段落を書く「ナレーター」** に絞った。Claude Code 系や MCP エージェントといった CLI/MCP 型主流とは別の領域で、対立ではなく役割分担として設計している。

本資料は LLM 開発者向けに、その判断の中身を **(1) 入出力 (2) 設計空間における選択 (3) 主流との関係 (4) やる／やらない** の四点で解説する。ネットワーク経路や鍵管理など API 呼び出しの実装プラミングは主題ではないため割愛。

---

## 1. 入出力

### 入力

- 当日のワークアウト数値（種目・距離・時間・標高ゲイン・心拍・ペース、HealthKit 由来）
- 当日の気象（気温・降水・風など）
- 中期コンテキスト：週次／月次のトレンド、ストリーク日数

意図的に **異なる軸（運動指標 × 気象 × 中期トレンド × 継続性）** を同時に渡す。**複数の横断した情報から傾向を掴む** のが LLM の得意領域なので、入力をヘテロに重ねる方向に振っている。

### 出力

- 自然言語の一段落
- 統計画面の「今日のサマリー」上に AI コーチからの一言として表示
- 本人の数値推移に紐づく具体（「直近3週は安定」「次のロング走は心拍を抑えめに」）まで踏み込む

---

## 2. 設計空間における選択

フィットネス × LLM の代表的な型を、比較のために並べる:

| 型 | 役割 |
| --- | --- |
| コーチ型チャット | 質問できる対話相手 |
| プランナー型 | ワークアウト計画を生成 |
| エージェント型 | カレンダー・通知と連動して走る予定を組む |
| フォーム解析型 | 動画から姿勢を指摘 |
| **ナレーター型** ← Cranq | 走った後に意味付けの一段落を書く |

ナレーター型を選んだ理由:

- **得意領域に乗せている** — 構造化された数値群を跨いで傾向を掴み自然文に落とすタスクは LLM が向きやすい。ルールベースで「ペース上昇 × 心拍抑制 × 3週連続」のような複数軸の組み合わせ解釈を書こうとすると条件分岐が膨れる
- **被害コストが小さい** — 過去解釈なら間違っても「文章として変」で済む。プラン生成だと「無理なメニュー指示」で怪我のリスク
- **評価軸が単純** — ワークアウト1件 → 段落1つ。入出力が単発なので回帰評価しやすく、対話履歴も多段思考も持たないので状態管理が要らない

---

## 3. 主流（CLI/MCP 型）との関係

2026 年現在、LLM アプリの主流は **CLI/MCP 型** ─ Claude Code、Cursor、Cline、各種 MCP エージェント。ユーザーが指示を出し、LLM がツールを使って能動的に目標を達成する形式。

ナレーター型はこれと **同じ土俵にいない**。対立ではなく役割分担として捉えている。

| 観点 | CLI/MCP 型 | ナレーター型 |
| --- | --- | --- |
| 起動契機 | ユーザーの指示 | 統計画面を開いた瞬間（暗黙起動） |
| 対話モード | ラリー（ユーザー ⇄ LLM 反復） | 一方通行（データ → LLM → 表示） |
| LLM の動作 | 能動（ツール呼び出し・多段推論） | 受動（与えられたデータを解釈） |
| ユーザーの役割 | ドライバー（指示する側） | 読み手（読む側） |
| 目標設定 | 動的（ユーザーが定義） | 固定（過去ワークアウトの意味付け） |

向く問題が異なる:

- 能動的なタスク遂行（コードを書く・調べる・編集する）→ CLI/MCP 型
- 受動的な情報咀嚼・意味付け（過去データを読みやすくする）→ ナレーター型

走り終わって統計画面を開いた瞬間に意味付けが添えられている、という体験は CLI/MCP の流儀では出せない。逆も然り。**両者が並走するのが LLM 時代の自然な姿** だと考えている。

## 4. やる／やらない

| 判断 | 内容 | 理由 |
| --- | --- | --- |
| ✅ やる | 過去解釈／ナレーション | 得意領域、被害コストが小さい |
| ✅ やる | 複数軸の横断入力（運動指標 × 気象 × 中期トレンド × 継続性） | 軸を跨いだ傾向の言語化を引き出すため |
| ❌ やらない | 未来プラン生成（「来週これを走れ」） | 運動指導領域でのハルシネーション被害を回避 |
| ❌ やらない | 対話ループ（チャット相手化） | ナレーション役に絞ることで評価軸を単純に保つ |
| ❌ やらない | エージェント化（ツール呼び出し・多段推論） | 「workout 1件 → 段落1つ」のスコープを閉じる |
| ⚠️ 割り切り | 出力のばらつきを許容 | 振り返り用途では文体多様性は欠点ではない |

CLI/MCP 型の追加余地としては「対話インタフェース」「ワークアウトを読める MCP サーバ化」「目標達成相談」が考えられる。ただし現状はやらない。仮に足すならナレーター型と並走する **別機能** として位置付ける（置き換えではない）。

---

## 5. 端的に

- **立場**: フィットネス × LLM で **ナレーター型** を選択。能動的エージェントではなく、走った後の意味付けに絞る
- **入力の妙**: 運動指標 × 気象 × 中期トレンドを同時に渡し、軸を跨いだ傾向を引き出す
- **主流との関係**: CLI/MCP 型（Claude Code 系・MCP エージェント）とは住み分け。対立ではなく役割分担
