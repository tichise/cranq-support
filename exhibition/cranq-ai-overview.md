# Cranq — LLM をどう組み込んでいるか／LLM で体験がどう変わったか

生成AIなんでも展示会向け。LLM 開発者・実装者に向けて、Cranq を **(1) LLM をどう組み込んでいるか（役割・入出力・スコープ）** と **(2) LLM 導入で体験がどう変わったか** の二点に絞って解説します。

ネットワーク経路や鍵管理など API 呼び出しの実装プラミングは本資料の主題ではないため割愛します。

---

## 1. LLM をどう組み込んでいるか

### 1.1 役割の定義

LLM は **「振り返りのナレーター」** として置いている。

- ✅ **過去解釈**: 走り終わったワークアウトに意味付けの一段落を書く
- ❌ **未来計画**: 「来週このメニューを走れ」は返さない
- ❌ **対話ループ**: チャット相手にしない（質問→回答のラリーを設計しない）
- ❌ **エージェント化**: ツール呼び出しや多段推論はしない

LLM の最も歩留まりが高い領域 — **構造化された数値からの自然文生成（要約・意味付け・文体付与）** — にスコープを絞っている。

### 1.2 入力（LLM に何を見せているか）

LLM が読むのは **HealthKit 由来の運動指標（数値）** と、それを文脈化するための補強情報。

- **当日のワークアウト**: 種目（サイクリング／ランニングなど）・距離・時間・標高ゲイン・平均/最大心拍・ペース
- **当日の天気**: 気温・降水・風など（Open-Meteo を端末から呼んで合流）
- **中期コンテキスト**: 週次／月次のトレンド、ストリーク日数
  - これがあるおかげで「今週は安定」「先月より落ちている」のような **時間軸を跨ぐ解釈** ができる
- **入れないもの**: 氏名・連絡先・端末識別子などの個人識別情報

### 1.3 出力（LLM に何を書かせているか）

- **形式**: 自然言語の一段落（短）
- **配置**: 統計画面の「今日のサマリー」上に表示
- **狙う文体**: 「励ます／指摘する／注意する」を自然な日本語で書き分け
- **個別性**: 本人の数値推移に紐づく文。テンプレ応答ではなく「直近3週は安定」「次のロング走は心拍を抑えめに」のような具体まで踏み込む
- **長さ**: 統計画面に違和感なく載るサイズ

### 1.4 スコープの設計

- **「ワークアウト1件 → 段落1つ」** の一発勝負にスコープを閉じている
- 対話履歴も多段思考も持たせない
- 評価軸を「出力された段落の質」だけに絞れる（=回帰テストもしやすい）
- 失敗しても被害が小さい（プラン生成と違って間違った指示で怪我させない）

### 1.5 アプリ内での位置づけ：LLM はサブシステム

- 履歴・統計・地図・グラフ・ストリーク・ウィジェットは LLM ゼロでフル機能動作
- LLM 経路が落ちても／返答が壊れても本体機能は無影響
- ユーザーがオプトインしない限り LLM 経路は起動しない
- 「LLM がなくても成立するアプリに、LLM で意味付けの層を被せる」設計

## 2. LLM で体験がどう変わったか

### 2.1 Before / After

| 観点 | LLM 以前のフィットネスアプリ | Cranq（LLM 以後） |
| --- | --- | --- |
| **数値の意味** | 距離・時間・心拍が並ぶだけ。解釈は本人任せ | 「今週はランの距離が安定」など、**意味が文として統計画面に返る** |
| **継続のフック** | グラフを自分で読み解く必要あり、数週で挫折しがち | ストリーク日数に LLM の言語化が重なり、続いていることが文でも読める |
| **意味付けの密度** | 振り返りは月1〜数回（気力次第） | ワークアウトごとに自動で一段落が添えられる |
| **解釈の主体** | 数字 → 本人が解釈 → 行動 | 数字 → LLM が解釈の叩き台 → 本人が読む |

### 2.2 何が新しく可能になったか

- **「数値」と「文」の同居** — 統計画面の「今日のサマリー」エリアに数字と LLM の一文が並ぶ。グラフだけでは届かなかった解釈が、同じ画面で読める
- **個別の継続フィードバック** — 「7日連続！」と祝うだけでなく **何が安定していて何が落ちているか** を文で返す。ストリーク機能が情緒的に立体化した
- **ロングテール文脈の織り込み** — 当日の天気・標高・心拍を LLM が拾うので「雨の中なのにこのペース、よくやった」のような **状況依存の励まし** が出せる土壌になる
- **意味付けが「無料の標準動作」になる** — 走り終わるたびに自分で振り返りを書かなくても、ワークアウトごとに一段落が添えられる。ユーザーは「書く」から「読む」に役割が変わる

### 2.3 LLM がなければ実現できなかった部分

- **テンプレ文では届かない個別性**
  - 「距離が伸びましたね」程度ならルールベースでも書ける
  - だが「直近3週はペースを上げたが心拍は抑えめ。フォーム改善が効いている可能性」のような **複数指標を跨ぐ解釈** はテンプレ化不能
- **語彙と文体の自然さ**
  - 「励ます／指摘する／注意する」を自然な日本語で書き分けるのはルールベースだとすぐ嘘くさくなる
  - LLM 以降、ここが量産可能になった
- **状況依存のトーン調整**
  - 雨の日の短い距離、暑い日の心拍上昇、休み明けの再開など、文脈に応じて文の構え方が自動で切り替わる

### 2.4 LLM 採用で諦めたこと・割り切ったこと

- **応答のばらつきを許容** — 同じワークアウトでも文は毎回少し違う。「振り返り」用途では多様性はむしろ歓迎
- **プラン生成はやらない** — 「来週このメニュー」を返さない。運動指導領域でハルシネーションが被害を出すリスクを避け、**過去解釈 → 自然文** に絞った
- **会話ループはしない** — チャット相手にしない。「ナレーション役」に徹することで評価軸（出力の質）を1次元に保つ
- **オフライン非対応** — LLM 経路はネット必須。本体機能（記録・閲覧・統計）はオフラインで動く設計でこれを補う

## 3. 端的に（A4プレート補強・口頭用）

- **LLM の使い方**: HealthKit のワークアウト数値＋週次トレンド＋天気 → 自然言語の一段落。対話せず、計画もせず、**振り返りに絞る**
- **体験変化**: 数値の羅列だった統計画面に「意味の一段落」が同居する。振り返りを自分で書かなくても、ワークアウトごとに解釈の叩き台が添えられる
- **割り切り**: 「ワークアウト1件 → 段落1つ」だけを LLM に任せる。汎用エージェント化を意図的に避けている
