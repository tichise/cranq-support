# Cranq — アーキテクチャと、LLM で体験がどう変わったか

生成AIなんでも展示会向け。LLM 開発者・実装者に向けて、Cranq を **(1) アーキテクチャ** と **(2) LLM 導入で体験がどう変わったか** の二本柱で解説します。マーケ説明ではなく、設計判断と Before/After を中心に書いています。

---

## 1. アーキテクチャ

### 1.1 データフロー（端末で完結）

```
HealthKit ──┐
            ├─→ 端末側で集約・派生値抽出 ──→ 構造化ペイロード ──→ OpenAI / Anthropic API ──→ 自然言語フィードバック ──→ 統計画面に表示
Open-Meteo ─┘                                                     ↑
                                                                  ユーザーの API キー（端末ローカル保存）
```

すべての段が iPhone 上で完結。Cranq 側のサーバー・プロキシ・ログストアは存在しない。

### 1.2 コンポーネント

- **データレイヤー**: Apple HealthKit
  - ワークアウト・GPS ルート・心拍・標高・距離・時間
  - Cranq は独自 DB を持たず、HealthKit を一次ストアとしてそのまま使う
- **コンテキスト補強**: Open-Meteo（履歴気象 API）
  - アクティビティの日時・地点から当日の天候を取得
  - 端末から直接呼び、結果は LLM 入力に合流
- **プロンプト整形層**: 端末ネイティブ Swift
  - 生 GPS ポリラインは渡さず、距離・標高ゲイン・平均/最大心拍などの **派生値だけ** を抽出
  - 氏名・連絡先・端末識別子は入れない
- **LLM 層**: OpenAI / Anthropic（ユーザー選択）
  - URLSession から直接 HTTPS。中継 SDK のテレメトリにも乗らない
- **鍵管理**: 端末ローカル保存のみ
  - Cranq のサーバー・クラウドに鍵を送る経路自体が実装されていない
- **出力**: 自然言語の段落
  - 統計画面の「今日のサマリー」上に表示
  - Cranq バックエンドに保存しない（バックエンドが存在しない）

### 1.3 設計上のキー判断

- **BYOK（Bring Your Own Key）** — アプリ運営側で LLM 一括契約しない。課金はユーザーの API 従量にそのまま乗る
- **中継サーバーなし** — iPhone → プロバイダ直送。Cranq 経路にプロンプトログが溜まらない構造
- **マルチプロバイダ** — OpenAI / Anthropic 両対応。新モデルが出たら鍵を差し替えるだけで反映できる（アプリのリリースサイクルに縛られない）
- **AI はサブシステム** — 閲覧・統計・ストリーク・ウィジェットは LLM ゼロでフル機能。AI 経路が落ちても本体は無影響
- **オプトイン徹底** — 鍵未設定なら LLM 経路は一切起動しない
- **送信は明示トリガ** — バックグラウンドで勝手に送らない。ユーザー操作で初めて発火

### 1.4 プロンプト設計

- **入力は構造化ペイロード**: 数値の塊を渡し、自然文への変換を LLM に丸投げする
- **個別性**: ユーザーごとの数値推移（週次・月次トレンド、ストリーク日数）が毎回コンテキストに乗るのでテンプレ応答にならない
- **出力はナラティブ単発**: 対話ループではなく「ワークアウト1件 → 段落1つ」の一発勝負
- **スコープを意図的に狭める**: 「HealthKit ワークアウト1件 → 自然言語1段落」だけを LLM に任せ、汎用エージェント化を避けている

## 2. LLM で体験がどう変わったか

### 2.1 Before / After

| 観点 | LLM 以前のフィットネスアプリ | Cranq（LLM 以後） |
| --- | --- | --- |
| **数値の意味** | 距離・時間・心拍が並ぶだけ。解釈は本人任せ | 「今週はランの距離が安定」など、**意味が文として返る** |
| **継続のフック** | グラフを自分で読み解く必要あり、数週で挫折しがち | LLM が「続いていること」を毎回言語化、続ける動機になる |
| **アドバイザー** | 人間コーチ（高額・予約必須・毎ライドは無理） | ワークアウトごとにオンデマンド、原価は API 数銭〜十数銭 |
| **シェア用文章** | 「今日のライド最高でした」と自分で書く | LLM が当日の数値を踏まえた一段落を生成 |
| **振り返り頻度** | 月1〜数回（気力次第） | ワークアウトごと（ボタン1つ） |
| **多言語対応** | 翻訳者が文を書く | 同じプロンプトで日英を両出し |

### 2.2 何が新しく可能になったか

- **「数値」と「文」の同居** — 統計画面に数字と LLM の一文が並ぶ。グラフだけでは届かない解釈が、同じ画面で読める
- **個別の継続フィードバック** — 「7日連続！」と祝うだけでなく **何が安定していて何が落ちているか** を文で返す。ストリーク機能が情緒的に立体化した
- **ロングテール文脈の織り込み** — 当日の天気・標高・心拍を LLM が拾うので「雨の中なのにこのペース、よくやった」のような **状況依存の励まし** が出る
- **「日記の代筆」** — 続かないワークアウト日記を LLM が肩代わり。ユーザーは「読む」側に回れる
- **コーチアクセスの民主化** — 月数千円のパーソナルコーチに頼まなくても、API 従量分（数十円〜数百円/月）で似たフィードバックが手に入る

### 2.3 LLM がなければ実現できなかった部分

- **テンプレ文では届かない個別性**
  - 「距離が伸びましたね」程度ならルールベースでも書ける
  - だが「直近3週はペースを上げたが心拍は抑えめ。フォーム改善が効いている可能性」のような **複数指標を跨ぐ解釈** はテンプレ化不能
- **語彙と文体の自然さ**
  - 「励ます／指摘する／注意する」を自然な日本語で書き分けるのはルールベースだとすぐ嘘くさくなる
  - LLM 以降、ここが量産可能になった
- **多言語の同時出力**
  - 同じ構造化ペイロードから日本語・英語の文を両方生成できる。各国語の文を人間が書かなくていい

### 2.4 LLM 採用で諦めたこと・割り切ったこと

- **応答のばらつきを許容** — 同じワークアウトでも文は毎回少し違う。だが「振り返り」用途では多様性はむしろ歓迎
- **プラン生成はやらない** — 「来週このメニュー」を返さない。運動指導領域でハルシネーションが被害を出すリスクを避け、**過去解釈 → 自然文** に絞った
- **会話ループはしない** — チャット相手にしない。「ナレーション役」に徹することで評価軸（出力の質）を1次元に保つ
- **オフライン非対応** — LLM 経路はネット必須。本体機能（記録・閲覧・統計）はオフラインで動く設計でこれを補う

---

## 3. 端的に（A4プレート補強・口頭用）

- **アーキテクチャ**: HealthKit → 端末で派生値抽出 → OpenAI / Anthropic 直送 → 自然文。中継サーバーなし、鍵はユーザー保管
- **体験変化**: 数値の羅列だった画面に「意味の一段落」が同居する。コーチに頼らないと得られなかった解釈が、ワークアウトごとに自動で出る
- **割り切り**: 計画 AI はやらない。**過去解釈 → 自然文** に LLM の使い道を絞っている
