# Cranq — AI / LLM 活用まとめ（生成AIなんでも展示会向け）

来場者・LLM 開発者向けに、Cranq の AI 周りで「何ができるか」「LLM をどう使っているか」をまとめた資料です。プレート（`cranq-plate.pptx`）の補足や、口頭説明のカンペとしても使える構成にしています。

## 一行で

> **HealthKit に積もったあなたのワークアウトを、自分の API キーで動く LLM に読ませて「振り返り」を書かせる、iOS 向けフィットネスアプリ。**
> 計画する AI ではなく、**振り返る AI**。日記を続けられない人の、書いてくれる版。

---

## 1. アーキテクチャ：BYOK × 直送 × 中継サーバーなし

- **BYOK（Bring Your Own Key）設計** — ユーザーが自分の OpenAI / Anthropic API キーを設定。アプリ運営側が LLM 利用料を負担する SaaS モデルではなく、課金は完全にユーザーの API アカウント側で従量。
- **中継サーバーが存在しない** — リクエストは **iPhone → OpenAI / Anthropic 直送**。Cranq 側にプロキシ・ログ・キャッシュレイヤーが一切ない。MITM できる場所がない構造。
- **API キーは端末ローカル保存のみ** — Cranq 側のサーバーやクラウドへキーを送らない。鍵が漏れる経路がそもそも実装されていない。
- **マルチプロバイダ対応** — OpenAI と Anthropic を選択可能。LLM ロックインを避け、好きなモデル提供者で動かせる。
- **iOS ネイティブ実装** — Swift / iOS 26.2+。LLM 呼び出しもネイティブの URLSession 直叩きで、サードパーティの中継 SDK に依存しない（余計な経路を増やさない）。
- **HealthKit を「データレイヤー」に丸投げ** — Cranq 自体はデータベースを持たず、Apple HealthKit をストアとして使う。AI に渡すデータの一次ソースが OS 標準。

## 2. AI に渡しているもの／渡していないもの（プロンプト設計の哲学）

- **渡すのは「数値だけ」** — 氏名・連絡先・SNS ID・端末識別子は送らない。LLM が見るのは HealthKit 由来の運動指標（距離・時間・標高・心拍・ペース・天気）。
- **位置情報は端末側で集約してから渡す** — 生の GPS ポリラインを丸ごと送りつけるのではなく、ルート由来の派生値（標高ゲイン、平均/最大心拍など）として要約。
- **天気は Open-Meteo から端末で取得** — 当日の気象も「数値」として LLM に渡す。降雨・気温などのコンテキストを与えることで、フィードバックを「今日の状況に当てた」ものにできる。
- **オプトイン徹底** — API キーが設定されない限り LLM は一切呼ばれない。アクティビティ閲覧・統計・ストリーク・ウィジェットは AI ゼロでフル機能動作する（AI を切っても劣化しない設計）。
- **送信はユーザー操作トリガ** — バックグラウンドで勝手に LLM へワークアウトを流さない。「コーチに読ませる」明示動作で初めて発火。

## 3. ポジショニング：「計画する AI」ではなく「振り返る AI」

- **トレーニングプラン生成 AI ではない** — 「来週このメニューを走れ」と命令するのではなく、**走った後に意味づけする** 役回り。
- **継続のための AI** — ストリーク・週次/月次チャートと組み合わさり、「続いていること」を言語化して返してくれる。続かない人の脱落点（自己満足の言語化が面倒）を肩代わりする。
- **日記の代筆者** — 「日記を続けられない人の、書いてくれる版」。LLM の自然言語生成能力を、運動の振り返り文に向けて使い切る使い方。
- **対話よりナレーション** — チャット相手というより、ワークアウトログにキャプションを付けてくれる存在。LLM の最も得意な領域（要約／文体／意味付け）にスコープを絞っている。
- **個別性のあるフィードバック** — 「今週はランの距離が安定しています。次のロング走は心拍を抑えめに」のような、本人の数値推移に紐づく文。テンプレ応答ではない。

## 4. プライバシー / コストの実装上の効果

- **数値だけを渡す** — 氏名や連絡先は送らない。AI に渡すのは HealthKit の運動指標。
- **BYOK = コスト透明性** — AI 利用ぶんはユーザーの OpenAI / Anthropic 従量請求にそのまま乗る。サブスクや追加課金ゼロ。アプリ運営者がリクエスト数を絞る動機がない（ペース制限を内側でかける誘惑がない）。
- **中継サーバーなし = ログ責任が分散** — Cranq 側にプロンプトログが溜まらない。データ流出の対象になる「Cranq 社内ログ」が物理的に存在しない。
- **AI はオプトイン** — 鍵を設定しない限り AI は動かない。記録閲覧と統計は単独で完結。
- **HealthKit はサーバーに送らない** — ワークアウトデータ本体は AI 利用時の OpenAI／Anthropic への送信ぶん以外、端末を一歩も出ない。

## 5. LLM 好きに刺さりそうな細部

- **「自分の鍵で、自分のデータを、自分のモデルに食わせる」** という三点セットがそのまま実装されている、数少ないコンシューマ iOS アプリ。
- **モデル選択の自由** — GPT 系か Claude 系かをユーザーが選ぶ。新モデルが出たら自分でキーを差し替えるだけで反映できる（アプリ側のリリースサイクルに縛られない）。
- **プロンプトの入力が「構造化された運動指標」** — 自然文ではなく数値ペイロードからの要約タスク。LLM の中でも「ナラティブ生成」を素直に評価できる題材。
- **失敗してもアプリが死なない** — LLM 出力が空でも壊れても、統計画面・ルート・天気・ストリークは独立して機能する。AI を「サブシステム」として疎結合に置いている良い見本。
- **小さく閉じた領域での LLM 活用例** — 巨大な汎用エージェントではなく、「HealthKit ワークアウト1件→自然言語1段落」というスコープを徹底的に狭めた応用。展示会で語るときの「LLM をどう使い倒すか」のミニケーススタディとして見せやすい。
- **OS 機能との合わせ技** — HealthKit + Apple Watch + iCloud + LLM。Apple のプライバシーモデル（HealthKit がアプリ越境を許さない）と、BYOK モデル（鍵・送信先をユーザー所有）が同じ思想で揃っている。

## 6. 来場者向けの一言キャッチ

- 「**鍵はあなたのもの。中継サーバーなし。**」（実装ポリシーをそのまま標語に）
- 「**走った記録を、AI と一緒に読み返す。**」（プレート主見出しと整合）
- 「**Your workouts, in your words — by your key.**」（英語）
- 「**計画する AI ではなく、振り返る AI。**」（ポジショニング）
