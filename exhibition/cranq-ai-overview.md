# Cranq — LLM の組み込み方と設計上の選択

Cranq は workout サイクルの **3つの起動点** で LLM を使う。**運動前 / 運動後 / 統計タイミング** で、それぞれ異なる時間軸の数値・文脈から自然言語の解釈を引き出している。

本資料は LLM 開発者向けに、その判断の中身を **(1) 三つの起動点 (2) 同じ抽象 I/F** の2点で解説する。ネットワーク経路や鍵管理など API 呼び出しの実装プラミングは主題ではないため割愛。

---

## 1. 三つの起動点

3 起動点は入力・出力・時間軸がそれぞれ違うが、共通して **複数の横断した情報から傾向を掴み自然文に落とす** という LLM の得意領域に乗っている。

### 1.1 運動前 — コンディション提示

- **入力**: 直近の運動強度（過去ワークアウトのトレンド）× 当日の天気 × 風速 × UV
- **出力**: 走り出す前の状況統合された一段落
- **時間軸**: 当日の数時間先
- **狙い**: 「直近強度高め × 風強い × UV 高い → 軽めに」のような **状況統合判断** を自然文で先出しする

### 1.2 運動後 — 個別ワークアウトの分析

- **入力**: 当日のワークアウト数値（種目・距離・時間・標高ゲイン・心拍・ペース）× 当日の気象 × 中期トレンド × ストリーク
- **出力**: 走った1件の意味付け一段落（統計画面の「今日のサマリー」）
- **時間軸**: 走り終わった直後
- **狙い**: 「直近3週は安定」「次のロング走は心拍を抑えめに」のような、本人の数値推移に紐づく **個別の詳細分析**

### 1.3 統計タイミング — 中長期の傾向分析

- **入力**: 習慣と傾向（週次・月次トレンド、ストリーク、種目バランス、強度分布）
- **出力**: **特徴の洗い出し → 原因の仮説 → 具体的なアクションのアドバイス** の三段構造
- **時間軸**: 数週〜数ヶ月
- **狙い**: ワークアウトがある程度溜まった統計画面で、単発を越えた中長期視点で観察 → 仮説 → 行動提案を出す

## 2. 同じ抽象 I/F

3 起動点は表面的には別の機能だが、実装上は **同じ抽象インターフェース** に乗っている:

```
入力: ヘテロな数値軸の束（運動指標・気象・UV・中期トレンド・継続性）
  → LLM
  → 出力: 自然言語の段落
```

3 機能を別々のロジックで書くのではなく、**入力軸の組み合わせを変えるだけ** で 3 つの体験を出している。

これが成立している理由:

- **LLM の得意領域に乗っている** — 構造化された数値群を跨いで傾向を掴み自然文に落とすタスクは LLM が向きやすい。ルールベースで「直近強度 × UV × 風速」「ペース上昇 × 心拍抑制 × 3週連続」のような複数軸の組み合わせ解釈を書こうとすると条件分岐が膨れる
- **評価軸が単純** — 起動点ごとに入出力が独立しているので、3点を別々に回帰評価できる。対話履歴も多段思考も持たないので状態管理が要らない

---

## 3. 端的に

- **3つの起動点**: 運動前（コンディション提示）／運動後（個別詳細分析）／統計タイミング（傾向分析・原因仮説・行動アドバイス）
- **同じ抽象 I/F**: ヘテロな数値軸の束 → LLM → 自然言語の段落、という共通の I/O。入力軸の組み合わせを変えるだけで 3 起動点を実現している
- **入力の妙**: 運動指標 × 気象・UV × 中期トレンドを重ね、軸を跨いだ傾向を自然文に落とす
